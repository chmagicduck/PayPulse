# 静态页反推 PRD 工作流

## 目的

把高保真静态页面转译成可开发、可验收、可维护的版本业务文档，避免逻辑开发直接建立在 mock 数据和演示状态上。

## 事实源优先级

1. `miniprogram/features/` 中已经还原完成的静态页面
2. `app.json`、`miniprogram/config/routes.ts`、现有 storage key 和共享工具
3. `docs/prototype/<version>/` 中的原型输入

规则：

- 静态页面是 PRD 阶段的主事实源
- 原型只在页面事实不足时作为补充
- mock 数字和示例文案不能直接当成真实业务事实

## 五步流程

### 1. 盘点页面事实

逐页记录：

- 页面目标
- 区块结构
- 局部状态
- 弹层和筛选态
- 已显式出现的字段和列表

### 2. 盘点共享线索

统一检查：

- 路由入口和返回链路
- storage key
- feature `model.ts`
- 跨页重复出现的文案、数字、卡片和列表

### 3. 抽取共享模型

把跨页重复概念升级为共享实体，例如：

- 用户配置
- 每日记录
- 时间轴事件
- 等级与任务进度
- 派生报表视图

### 4. 补齐闭环假设

当页面事实不足以支撑开发时，必须显式补齐：

- 启动流程
- 初始化规则
- 清空与重置流程
- 计算口径
- 空态与异常态

所有补齐内容都要标注为 `闭环假设`。

### 5. 输出版本 PRD

固定输出：

- `README.md`
- `overview.md`
- `glossary.md`
- `domain-model.md`
- `data-model.md`
- `page-map.md`
- `acceptance.md`
- `pages/*.md`

## PRD 完成标准

- 所有注册页面都有页面文档
- 所有共享规则都集中在 `domain-model.md`
- 所有共享字段和 key 都集中在 `data-model.md`
- 启动、初始化、清空和关键派生链路是闭环的
- 逻辑开发者不需要再从静态页自行猜核心规则

## 反模式

- 直接照抄 mock 数字当成真实业务
- 在每个页面文档里重复抄共享规则
- 跳过启动和重置流程，只写 happy path
- 不区分 `页面直接事实`、`跨页推理`、`闭环假设`
