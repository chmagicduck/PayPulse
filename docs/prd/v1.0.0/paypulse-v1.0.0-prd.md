# PayPulse v1.0.0 产品需求文档

- 版本：`v1.0.0`
- 文档状态：正式版
- 产出日期：`2026-03-26`
- 适用范围：微信小程序 `PayPulse`
- 基线来源：`miniprogram/features` 当前实现、`docs/prototype/1.0.0` 原型稿、已确认的 v1.0.0 功能拆解计划

## 1. 产品目标与版本边界

### 1.1 产品定位

PayPulse 是一款面向 25-30 岁职场用户的微信小程序，用航海隐喻表达工作、摸鱼、人生阶段和日常成就。  
`v1.0.0` 的目标不是做完整运营体系，而是先完成以下闭环：

1. 用户首次进入后，能够完成基础档案设置并顺利进入首页。
2. 首页、战报、动力舱、个人基地使用同一套真实业务口径，不再依赖演示数据。
3. 岁月坐标、工作日历、动力舱成长、社区入口形成基础可用版本，支持后续迭代。

### 1.2 目标用户

- 年龄主区间：`25-30 岁`
- 典型特征：
  - 已进入稳定工作阶段
  - 对收入、工时、休息节奏较敏感
  - 乐于接受轻松、幽默、游戏化表达
  - 不愿承担复杂配置成本，希望首次设置低阻力

### 1.3 v1.0.0 成功标准

- 首次使用到进入首页形成完整闭环，无可绕过路径。
- 首页、战报、个人基地、动力舱之间的数据口径一致。
- 所有关键数值均可被追溯到明确数据源和公式。
- 社区入口可真实使用，而不是仅保留静态视觉。

### 1.4 当前代码基线

当前工程已经具备以下基础：

- 已存在 `AppBootstrapState` 初始化闭环，未完成设置时会强制跳转到设置页。
- 设置页已有基础表单、保存逻辑、初始化逻辑。
- 岁月坐标页已有固定档案本 UI，以及新增、编辑、删除条目能力。
- 动力舱、首页、战报、个人基地已有原型级页面结构和部分存储设计。
- 数据中心已有导出和清空能力。
- 社区页已有静态视觉页，示例目录提供了真实企微入群方案。

当前工程仍存在以下问题，需要在研发阶段按本 PRD 重构：

- 首页和战报仍混用演示口径与半动态口径。
- `DailyVoyageRecord` 目前仍带有伪随机种子生成逻辑，不符合真实摸鱼会话累计。
- 动力舱成就口径与产品要求不一致，部分进度来自静态或错误推导。
- 战报最近记录当前保留了可编辑思路，不符合“领潮中心唯一修改入口”的产品约束。
- 社区页当前仅保留静态展示，未接真实入群流程。
- 日历当前完整法定节假日能力仅覆盖 `2026` 年，且未抽象为多年份扩展模型。

### 1.5 版本范围

#### 本版本必须完成

- 首次设置闭环
- 岁月坐标系统卡与用户卡规则
- 首页真实摸鱼/航行口径
- 动力舱每日任务、每周成就、等级曲线
- 战报真实数据读取与联动修正
- 个人基地累计摸鱼打卡依据
- 真实企微社区接入
- 可扩展的多年份日历架构

#### 本版本明确不做

- 其他年份法定节假日数据补录
- 云端同步与跨设备同步
- 自动识别喝水、起身、视线训练等外部行为
- 首页直接完成动力舱任务打卡
- 用户自定义档案本

## 2. 共享数据模型

本节定义 `v1.0.0` 的唯一数据源关系。研发不得再以页面静态 `model.ts` 作为正式业务事实来源。

### 2.1 数据源总览

| 模型 | 作用 | 唯一来源 | 主要消费者 |
| --- | --- | --- | --- |
| `AppBootstrapState` | 控制是否完成首轮设置 | 本地存储 | app 启动、设置页、数据中心 |
| `ProfileSettings` | 用户基础档案 | 本地存储 | 设置页、首页、日历、岁月坐标、战报、人生航程 |
| `TimeAxisEntry` | 岁月坐标条目 | 本地存储 | 岁月坐标页、首页、日历 |
| `DailyVoyageRecord` | 每日工时/摸鱼事实记录 | 本地存储 | 首页、战报、个人基地 |
| `LabProgress` | 动力舱成长进度 | 本地存储 | 动力舱页、首页任务预览、个人基地等级展示 |
| `MoyuSession` | 当前进行中的摸鱼会话 | 本地存储 + 内存态 | 首页、战报联动、冷启动恢复 |
| `CalendarYearConfig` | 指定年份日历配置 | 本地配置 | 日历、首页常规波动、工时计算 |

### 2.1.1 状态定义表

#### 首页状态枚举

| 状态 | 文案 | 是否允许开启摸鱼 |
| --- | --- | --- |
| `pre-work` | `待启航` | 否 |
| `working` | 距离下班剩余时间 | 是 |
| `lunch-break` | `午休中` | 否 |
| `off-duty` | `已离岗` | 否 |
| `rest-day` | `充电中` | 否 |

#### 摸鱼会话状态枚举

| 状态 | 含义 |
| --- | --- |
| `idle` | 当前无活跃摸鱼会话 |
| `active` | 当前存在活跃摸鱼会话 |
| `auto-stopped` | 活跃会话因时间边界或冷启动恢复被自动终止 |

### 2.2 现有模型沿用与补充

#### `AppBootstrapState`

沿用现有模型：

- `schemaVersion`
- `status: 'needs_setup' | 'ready'`
- `initializedAt`
- `lastResetAt`

规则：

- `status = needs_setup` 时，小程序首屏必须强制进入设置页。
- 数据中心清空数据后，必须回到 `needs_setup`。

#### `ProfileSettings`

沿用现有字段，并将其作为以下模块唯一输入：

- 首页收益计算
- 工作日历
- 人生航程看板
- 默认岁月坐标生成
- 动态退休时间计算

字段口径：

- `nickname`：用户昵称，必填
- `birthday`：生日，必填
- `gender`：性别，必填
- `careerStartDate`：就业纪念日来源，必填
- `retirementAge`：默认根据官方口径自动带出，可手改
- `expectedLifespan`：默认 `85`
- `monthlySalaryCents`：月薪，单位分，必填
- `payDay`：发薪日，默认 `10`
- `workMode`：`double | single-sat | single-sun | big-small`
- `isCurrentBigWeek`
- `startTime`
- `endTime`
- `lunchBreakEnabled`
- `lunchStartTime`
- `lunchEndTime`

#### `TimeAxisEntry`

现有模型需要补充以下字段：

- `sourceType: 'system' | 'user'`
- `systemType: 'birthday' | 'career-anniversary' | 'retirement-day' | null`
- `locked: boolean`

规则：

- 系统卡固定 3 张：
  - `我的生日`
  - `就业纪念日`
  - `退休日`
- 系统卡默认值固定为：
  - `我的生日`：`isAnniversary = true`
  - `就业纪念日`：`isAnniversary = true`
  - `退休日`：`isAnniversary = false`
- 系统卡必须具备：
  - `sourceType = system`
  - `locked = true`
  - 不允许在岁月坐标页删除
- 用户卡必须具备：
  - `sourceType = user`
  - `locked = false`
  - 支持新增、编辑、删除

#### `DailyVoyageRecord`

`v1.0.0` 调整为“存时间事实，不长期固化收益结果”的模型。  
推荐结构如下：

- `date`
- `calendarStatus`
- `scheduledWorkDurationSec`
- `moyuDurationSec`
- `manualAdjustmentDurationSec`
- `adjustmentSource`
- `updatedAt`

读取阶段动态派生：

- `actualWorkDurationSec = max(0, scheduledWorkDurationSec - moyuDurationSec)`
- `moyuIncomeCents`
- `voyageIncomeCents`
- `moyuRatio`

这样可以保证当用户修改工作模式、午休、上下班时间后，历史数据能按最新口径联动重算，不保留失真的旧收益。

额外约束：

- 不再持久化最终收益结果，避免旧口径污染历史记录。
- 若本地存量数据中存在 `derivedIncomeCents` 一类旧展示字段，读取时仅作为兼容残留字段处理，不再作为正式事实来源。
- 历史记录重算是产品明确选择，不引入设置快照版本模型。
- `moyuDurationSec` 必须满足 `0 <= moyuDurationSec <= 当日计划工作时长秒数`。
- 当用户修改工作模式、上下班时间、午休开关或午休时段，导致当日或历史某日计划工作时长下降时，超出上限的 `moyuDurationSec` 必须自动裁剪到新的当日上限。

#### `LabProgress`

沿用现有模型，保持：

- `totalPoints`
- `todayPoints`
- `selectedRankIndex`
- `lastDailyResetDate`
- `lastWeeklyResetDate`
- `tasks`
- `achievements`

补充规则：

- 每日任务进度只存真实打卡次数。
- 每周成就进度只存本周累计结果。
- 等级由 `totalPoints` 唯一反推，不允许页面自行维护。
- 个人基地中的头像外圈 tone、头像角标 icon、`Lv.x 等级名` 必须全部由 `LabProgress.totalPoints -> 当前等级` 唯一推导。
- 个人基地不维护独立等级，不允许手动设置，不允许与动力舱等级脱钩。

### 2.3 新增模型

#### `MoyuSession`

用于保证“开启摸鱼后切后台或回到页面不中断，杀进程后结束”。

建议字段：

- `sessionId`
- `status: 'idle' | 'active'`
- `date`
- `startedAt`
- `lastTickAt`
- `accumulatedDurationSec`
- `stopReason: 'manual' | 'cross-lunch' | 'off-duty' | 'holiday' | 'cold-start-recover' | null`

规则：

- 开启摸鱼时创建会话。
- 首页实时状态必须以 `MoyuSession + ProfileSettings + CalendarYearConfig` 联合判定，不允许页面本地临时变量自行推导业务状态。
- 前后台切换时根据时间戳补差，不中断会话。
- 跨午休、跨下班、休息日状态下必须自动结束或禁止开启。
- 冷启动时若发现遗留 `active` 会话，按 `lastTickAt` 结算后强制关闭，视为“杀进程后结束”。
- 自动终止后，需先将结算结果回写当天 `DailyVoyageRecord`，再将会话切为 `auto-stopped`。
- 用户显式关闭或手动修正今日摸鱼时长后，会话必须结束，不允许残留活跃状态。

#### `CalendarYearConfig`

建议字段：

- `year`
- `holidayRanges`
- `makeupDays`
- `supportOfficialHoliday: boolean`

规则：

- `2026` 年：`supportOfficialHoliday = true`，展示完整法定节假日与调休。
- 其他年份：`supportOfficialHoliday = false`，只按周末和工作制推导，不展示法定节假日与调休标记。
- 日历渲染层只消费 `CalendarYearConfig`，不得写死单年份逻辑在页面代码中。
- 首页常规波动卡片、工作时长计算、倒计时状态必须统一消费 `CalendarYearConfig + ProfileSettings` 的联合结果。

## 3. 页面与模块逻辑拆解

## 3.1 第一步：航行档案设置页与首次闭环

### 3.1.1 页面目标

让用户以最低成本完成基础档案设置，并建立整个小程序后续计算所需的唯一基础参数。

### 3.1.2 页面进入规则

- 小程序首次进入时，若 `AppBootstrapState.status = needs_setup`，直接进入设置页。
- 用户在数据中心执行“清空数据”后，再次进入也必须直接进入设置页。
- 未完成设置前：
  - 左上返回按钮保留视觉位置
  - 呈禁用态视觉
  - 不响应点击
  - 不允许通过系统返回、tab 切换、页面跳转绕过

### 3.1.3 首次默认值策略

采用“半预填低阻力”策略。

默认预填：

- `workMode = double`
- `startTime = 09:00`
- `endTime = 18:00`
- `lunchBreakEnabled = true`
- `lunchStartTime = 12:00`
- `lunchEndTime = 13:30`
- `payDay = 10`
- `expectedLifespan = 85`
- `retirementAge = 根据 gender + birthday + 官方口径自动带出`

必须要求用户确认或填写：

- `nickname`
- `gender`
- `birthday`
- `careerStartDate`
- `monthlySalary`

默认值设计原则：

- 不追求“示例值完整”，而追求“降低首次输入成本且不误导”。
- 与 25-30 岁用户常见工作时段相符。
- 午休默认打开，比默认关闭更合理，因为它直接影响摸鱼状态、收益口径和下班倒计时状态判断。

### 3.1.4 退休年龄默认规则

- 产品要求：根据 `gender + birthday` 自动填充默认退休年龄。
- 实现要求：
  - 使用“可配置规则表”或“配置驱动函数”
  - 不将政策口径硬编码在页面层
  - 后续政策变更时，只替换配置，不改 UI 逻辑
  - 表单态需额外维护 `retirementAgeEditedByUser` 或等价标记，用于区分“系统默认值”和“用户手输值”

交互规则：

- 用户首次进入时，选择性别和生日后自动带出默认退休年龄。
- 若用户未手动改动退休年龄，再修改性别或生日时，退休年龄同步刷新。
- 若用户已手动编辑退休年龄，后续性别或生日变化不得静默覆盖，需保留用户手输值。

### 3.1.5 保存后的初始化动作

首次保存成功后，必须按如下顺序完成初始化：

1. 写入 `ProfileSettings`
2. 写入默认头像
3. 生成系统岁月坐标 3 张卡
4. 生成当月 `DailyVoyageRecord`
5. 初始化 `LabProgress`
6. 清空遗留演示数据或兼容旧数据
7. 将 `AppBootstrapState.status` 标记为 `ready`
8. 自动进入首页

优先级说明：

- “清理旧演示数据或兼容旧数据”必须在正式业务数据读取前完成，避免旧演示字段污染首页与战报。
- “生成系统岁月坐标 3 张卡”是强制初始化动作，不得因为本地残留脏数据而跳过。

### 3.1.6 编辑设置后的联动规则

当用户已完成初始化后再次修改设置，保存后不再走首次初始化，而是触发联动重算：

- 首页收益与状态
- 战报趋势与占比
- 人生航程看板
- 日历工作日判定
- 系统岁月坐标日期

## 3.2 第二步：岁月坐标设置页

### 3.2.1 页面目标

作为“固定档案本 + 用户自定义人生事件”的轻量管理页面，支持管理重要日期，并为首页和日历提供提醒来源。

### 3.2.2 档案本规则

- 档案本固定为系统预设，不支持新增、删除、改名。
- 档案本仅承担分类作用。

系统档案本保留：

- 全部
- 纪念本
- 旅行本
- 人生本
- 职场本

### 3.2.3 条目规则

系统卡：

- `我的生日`
- `就业纪念日`
- `退休日`

系统卡行为：

- 自动生成
- 永久存在
- 不允许删除
- 日期和核心含义来自设置页

用户卡行为：

- 支持新增
- 支持修改
- 支持删除
- 支持选择是否开启“周年提醒模式”

### 3.2.4 首页读取规则

- 首页仅展示 `isAnniversary = true` 的条目。
- 按最近到来时间升序排序。
- 最多展示 3 张。
- 若不足 3 张，则展示实际数量，不补占位假数据。

边界条件：

- 系统卡中的 `退休日` 默认不进入首页岁月坐标卡片区，因为其 `isAnniversary = false`。
- 用户删除自建卡后，不影响系统卡存在性和排序逻辑。

## 3.3 第三步：动力舱与等级逻辑

### 3.3.1 页面目标

通过“每日可打卡任务 + 每周成就 + 长线等级成长”建立用户的持续反馈。

### 3.3.2 数据来源原则

- 每日任务：全部手动打卡
- 每周成就：由任务和业务记录推导
- 等级：由总动能唯一决定

### 3.3.3 刷新规则

- 每日任务：
  - 每天 `00:00` 后不主动定时刷新
  - 用户下一次打开小程序、进入首页或动力舱时判定重置
- 每周成就：
  - 每周一 `00:00` 后不主动定时刷新
  - 用户下一次打开小程序或进入动力舱时判定重置

### 3.3.4 每日任务设计

| 任务 | 单次动能 | 每日上限次数 | 当日最大动能 | 说明 |
| --- | --- | --- | --- | --- |
| 深海补水计划 | `+1` | `8` | `8` | 轻量高频 |
| 甲板体能回收 | `+1` | `10` | `10` | 轻量高频 |
| 远场海平线 | `+1` | `10` | `10` | 轻量高频 |
| 准点回港靠岸 | `+12` | `1` | `12` | 高价值单次 |
| 潜艇压力排放 | `+4` | `5` | `20` | 中频任务 |

- 日常任务当日总上限：`60`

### 3.3.5 每周成就设计

| 成就 | 跟踪对象 | 目标值 | 奖励 |
| --- | --- | --- | --- |
| 黄金航线守护者 | 准点回港靠岸 | 连续 `7` 天完成 | `+50` |
| 静默巡航专家 | 潜艇压力排放 | 单周累计 `120` 分钟 | `+50` |
| 满点核动力 | 深海补水计划 | 单周累计 `40` 次 | `+50` |

- 每周成就总上限：`150`

实现规则：

- 每日任务全部手动打卡，但每周成就必须由任务记录自动推导，不允许手动直接修改成就进度。
- 每周一重置成就周进度，不跨周累计。
- `黄金航线守护者` 的“连续 7 天”按当前周内连续自然日计算；若某天未完成则本周连续计数中断。

### 3.3.6 等级阈值

| 等级 | 目标动能 |
| --- | --- |
| Lv1 | `0` |
| Lv2 | `100` |
| Lv3 | `260` |
| Lv4 | `520` |
| Lv5 | `900` |
| Lv6 | `1400` |
| Lv7 | `2050` |
| Lv8 | `2900` |
| Lv9 | `3950` |
| Lv10 | `5200` |

设计原则：

- 活跃用户约 90 天达到满级。
- 前 3 级反馈要快，中后期逐步拉开。
- 每周成就是重要加速器，但不足以替代日常任务。

### 3.3.7 进度条口径

当前等级进度：

`(当前总动能 - 当前等级起点) / (下一等级目标 - 当前等级起点)`

规则：

- 结果范围固定在 `0% - 100%`
- 满级后始终显示 `100%`
- 首页动力舱任务预览和动力舱页进度条口径必须完全一致

## 3.4 第四步：首页功能拆解

### 3.4.1 页面目标

首页是核心驾驶舱，负责展示用户当天工作节奏、摸鱼状态、收入换算、重点提醒和人生航程。

### 3.4.2 核心名词定义

- `摸鱼 = 摸鱼`
- `航行 = 今日实际发生工时`

### 3.4.3 收益口径

以下首页收益全部按“截至当前时刻已发生的真实时间”计算，不按整日计划预支未来收益。

- `单位航速（秒薪） = 月薪分 / 当月计划工作总秒数`
- `已发生计划工时秒数 = 今日从上班开始到当前时刻之间的有效工作秒数 - 午休秒数 - 非工作时段秒数`
- `今日实际发生工时秒数 = max(0, 已发生计划工时秒数 - 今日累计摸鱼秒数)`
- `今日摸鱼收益 = 今日累计摸鱼秒数 * 秒薪`
- `今日航行收益 = 今日实际发生工时秒数 * 秒薪`
- `本月累计航行收益 = 本月每日实际发生工时秒数汇总 * 秒薪`

说明：

- `今日摸鱼收益` 是 `今日航行收益` 的子集。
- 今日和本月的总收益都不得重复加总摸鱼收益。
- 所有收益以“分”为最小精度存算，展示时再格式化为元。
- 若当前时刻早于上班时间，则 `已发生计划工时秒数 = 0`。
- 若当前时刻处于午休中，则只计算午休开始前已发生的有效工作时间，不把午休时间记入分母。
- 若当前时刻晚于下班时间，则“已发生计划工时秒数”封顶为当日完整计划工时秒数。

### 3.4.4 今日摸鱼收益刷新规则

- 仅在以下动作发生时刷新：
  - 开启摸鱼
  - 结束摸鱼
  - 手动修正今日摸鱼时长
- 不允许页面定时伪增长“摸鱼金额”。

修改入口约束：

- 用户只能在首页，即“领潮中心”，修改今日摸鱼时长。
- 战报页最近记录不提供编辑入口，不提供弹窗修改能力。
- 领潮中心对今日摸鱼时长的手动修正，必须实时回写 `DailyVoyageRecord`，并结束当前活跃 `MoyuSession`。
- 手动修正值不得超过“今日计划工作时长秒数”；若用户输入超限，必须自动裁剪到上限值后再保存。

### 3.4.5 摸鱼会话状态机

状态节点：

- `idle`
- `active`
- `auto-stopped`

状态转换：

1. 满足工作中条件时，用户点击“开启摸鱼”，`idle -> active`
2. 用户主动关闭摸鱼，`active -> idle`
3. 切后台、切页面、回前台，只更新时间戳，不改变 `active`
4. 跨午休边界、跨下班边界、进入休息日，`active -> auto-stopped`
5. 冷启动检测到遗留会话时，按最后有效时间结算并 `active -> auto-stopped`

### 3.4.6 首页状态定义

| 状态 | 触发条件 | 文案 | 是否允许开启摸鱼 |
| --- | --- | --- | --- |
| `待启航` | 工作日且当前时刻早于上班时间 | `待启航` | 否 |
| `工作中` | 工作日且位于有效工作时段 | 距离下班剩余时间 | 是 |
| `午休中` | 工作日且当前时刻位于午休区间 | `午休中` | 否 |
| `已离岗` | 工作日且当前时刻晚于下班时间 | `已离岗` | 否 |
| `充电中` | 休息日或法定节假日 | `充电中` | 否 |

### 3.4.7 摸鱼占比

公式：

`摸鱼占比 = 今日累计摸鱼秒数 / 今日计划工作秒数`

规则：

- 每天按自然日 `00:00` 清空
- 今日不是工作日时，不计算有效分母
- 进入上班时间后开始有效
- 下班后停止增长
- 若当日计划工时为 `0`，则占比显示 `0%`

### 3.4.8 离岗倒计时文案

| 场景 | 文案 | 是否允许开启摸鱼 |
| --- | --- | --- |
| 上班前 | `待启航` | 否 |
| 正常工作时段 | 距离下班剩余时间 | 是 |
| 午休时段 | `午休中` | 否 |
| 下班后 | `已离岗` | 否 |
| 休息日/法定假日 | `充电中` | 否 |

### 3.4.9 常规波动卡片

首页常规波动以日历为准：

- 发薪日
- 休息日
- 法定节假日
- 调休日

联动规则：

- 修改 `payDay` 后，发薪倒计时立即联动。
- 修改工作制后，休息日倒计时和日历同步联动。
- 非 `2026` 年不展示法定节假日/调休标签，只保留工作制和周末逻辑。
- 不允许页面直接写死 `2026` 常量来渲染首页卡片，必须统一使用 `CalendarYearConfig + ProfileSettings` 联合计算。

### 3.4.10 岁月坐标卡片

- 数据来源：`TimeAxisEntry`
- 仅展示 `isAnniversary = true`
- 按最近日期排序
- 最多展示 `3` 张

系统默认生成：

- `就业纪念日`
- `我的生日`

如用户存在更多开启周年提醒的卡片，则按最近原则参与排序。

联动关系：

- 修改设置页中的生日、入职日期后，首页岁月坐标卡片排序需立即刷新。
- 修改岁月坐标中用户卡的周年提醒开关后，首页展示结果需立即刷新。

### 3.4.11 人生航程看板

数据完全基于设置计算：

- 生命已走过时长：基于 `birthday`
- 职场已走过时长：基于 `careerStartDate`
- 距离退休：基于 `retirementAge`
- 距离生命终点：基于 `expectedLifespan`

禁止继续使用原型静态值。

## 3.5 第五步：战报页的数据依据

### 3.5.1 页面目标

将首页的单日状态扩展成趋势、占比与历史回看页面，作为只读分析页，不承担摸鱼时长编辑职责。

### 3.5.2 数据来源

战报所有核心数据统一来自 `DailyVoyageRecord`：

- 趋势图
- 占比环
- 年度卡
- 历史列表

### 3.5.3 趋势图规则

- 默认展示当前月
- 有几天真实记录就展示几天
- 完全无记录时显示空态说明
- 不允许补伪数据、不允许继续保留原型示例柱图
- 当天趋势柱只展示截至当前时刻的真实数据，不展示整日预测值

空态区分：

- `本月无数据`：当前月尚无任何真实记录
- `本月数据不足完整月`：当前月已有记录，但仅展示已有自然日，不补齐未来日期

### 3.5.4 历史修正规则

- 战报页最近记录移除编辑功能，不提供任何手动修改入口。
- 用户只能在首页“领潮中心”修改今日摸鱼时长。
- 首页完成今日摸鱼时长修正后：
  - 更新当日 `DailyVoyageRecord.moyuDurationSec`
  - 标记 `adjustmentSource = home`
  - 首页、战报、个人基地、动力舱相关联动视图全部重新读取
- 不支持在战报页直接修正历史任意日期的摸鱼时长。

设置修改后的历史策略：

- 用户修改月薪、工作制、上下班时间、午休开关、午休时段后，历史记录全部按最新档案联动重算。
- 这是产品明确选择，不引入设置快照或版本化历史口径。
- 若设置变化导致某日计划工作时长下降，则该日 `moyuDurationSec` 必须先被裁剪到新的工时上限，再参与收益和占比重算。

### 3.5.5 年度卡口径

推荐口径：

- 年度摸鱼收益：所有 `moyuIncomeCents` 汇总
- 摸鱼打卡天数：`moyuDurationSec > 0` 的自然日数量
- 月均摸鱼收益：年度摸鱼收益 / 有记录月份数
- 年度航行收益：所有 `voyageIncomeCents` 汇总

## 3.6 第六步：个人基地的累计摸鱼打卡依据

### 3.6.1 页面目标

展示用户身份信息、头像、导航入口和长期累计结果。

### 3.6.2 累计摸鱼打卡天数定义

`累计摸鱼打卡天数 = 历史上所有存在摸鱼记录的自然日数量`

判定标准：

- 只要当日 `moyuDurationSec > 0`，即记为 1 天打卡成功
- 不要求满 1 分钟，不要求满 5 分钟

- 等级、头像徽章、等级文案：`LabProgress.totalPoints` 推导出的当前等级

### 3.6.4 头像与等级联动规则

- 个人基地展示的 `Lv.等级 + 等级名`，必须与动力舱当前等级面板完全一致。
- 个人基地不允许维护独立等级快照，不允许保留静态 `user.rank` 作为业务事实来源。
- 当动力舱总动能变化导致升级或等级展示变化时，个人基地在下次进入页面或页面重新 `onShow` 时同步刷新。
- 清空数据或首次初始化后，个人基地默认等级必须与动力舱初始等级一致。

## 4. 公式与状态机

## 4.1 工作时长与日历

### 4.1.1 计划工作时长

`计划工作时长 = 下班时间 - 上班时间 - 午休时长`

规则：

- 法定假日、休息日：计划工作时长为 `0`
- 调休日、工作日：按完整工作制计算
- 任意自然日的摸鱼时长不得超过该日计划工作时长。

### 4.1.2 当月计划工作总秒数

`当月计划工作总秒数 = 当月每个自然日的计划工作时长累加`

### 4.1.3 实际发生工时

`今日实际发生工时秒数 = max(0, 已发生计划工时秒数 - 今日摸鱼秒数)`

### 4.1.4 秒薪

`秒薪 = 月薪分 / 当月计划工作总秒数`

### 4.1.5 收益

- `已发生计划工时秒数 = 今日从上班开始到当前时刻之间的有效工作秒数 - 午休秒数 - 非工作时段秒数`
- `今日摸鱼收益 = 今日摸鱼秒数 * 秒薪`
- `今日航行收益 = 今日实际发生工时秒数 * 秒薪`
- `本月累计航行收益 = Σ(每日实际发生工时秒数 * 当月秒薪)`

所有计算必须先在“分”和“秒”的维度完成，再做展示格式化，保证最终无误差。

补充规则：

- 上班前，`已发生计划工时秒数 = 0`
- 午休中，分母只统计午休前已发生的有效工作秒数
- 下班后，已发生计划工时秒数封顶为当日完整计划工时
- 所有历史记录在设置修改后按最新档案重算
- 在任何手动修正和联动重算过程中，`今日摸鱼秒数` 与历史 `moyuDurationSec` 都必须被裁剪到对应自然日的计划工作时长上限之内

## 4.2 设置页联动状态机

状态：

- `未初始化`
- `已初始化`

流程：

1. 启动应用
2. 若 `needs_setup`，进入设置页
3. 用户填写并保存
4. 首次保存时完成初始化
5. 进入首页
6. 后续再编辑时，只触发联动重算，不重置用户其他业务数据

### 4.3.1 每日重置

触发条件：

- 用户打开小程序
- 用户进入首页
- 用户进入动力舱

若检测到 `lastDailyResetDate != today`：

- 清空每日任务次数
- 清空 `todayPoints`
- 保留 `totalPoints`

### 4.3.2 每周重置

若检测到 `lastWeeklyResetDate != currentWeekMonday`：

- 重置每周成就进度
- 保留 `totalPoints`

## 4.4 多年份日历架构

### 4.4.1 2026 年

- 使用完整法定节假日与调休日配置
- 日历卡片、首页波动、工时计算均生效

### 4.4.2 其他年份

- 只保留以下能力：
  - 周末判定
  - 双休/单休/大小周判定
  - 发薪日计算
- 不展示：
  - 法定节假日标签
  - 调休日标签

### 4.4.3 扩展要求

- 后续新增年份时，只需要补充新年份配置，不需要改页面逻辑

## 4.5 实现接口与约束

- 首页、战报、个人基地的业务读取统一以 `DailyVoyageRecord` 为准，不允许页面保留独立收益口径。
- 首页实时状态统一以 `MoyuSession + ProfileSettings + CalendarYearConfig` 联合判定。
- 首页“领潮中心”是唯一允许修改摸鱼时长的页面入口；战报页最近记录为只读。

- 社区主按钮必须接真实企微能力；错误码语义沿用示例目录，不允许静态按钮伪装成可用功能。
- 文档中未显式授权的“页面本地推断逻辑”一律不视为正式业务口径。

## 5. 验收口径

## 5.1 首次闭环

- 首次进入必须进入设置页
- 清空数据后再次进入必须进入设置页
- 未初始化时返回按钮必须显式禁用
- 首次保存后必须自动进入首页

## 5.2 设置联动

- 修改性别、生日后，退休年龄默认值按规则更新
- 用户手动编辑过退休年龄后，不得被静默覆盖
- 修改工作模式、午休、上下班时间后，首页、日历、战报、人生航程必须同步联动

## 5.3 岁月坐标

- 系统卡必须始终存在
- 系统卡不可删除
- 用户卡增删改不影响系统卡
- 首页最多显示 3 张周年提醒卡

## 5.4 首页与战报

- 摸鱼会话切后台、回前台、切页面不中断
- 杀进程后重启不得继续累计旧会话
- 午休、下班、休息日状态下不可开启摸鱼
- 今日摸鱼收益必须小于等于今日航行收益
- 本月累计航行收益必须等于本月每日真实收益汇总
- 战报无数据时显示空态，不补假数据
- 战报最近记录必须为只读，不提供编辑入口
- 用户只能在首页“领潮中心”修改今日摸鱼时长
- 任意自然日的摸鱼时长不得超过该日总工时；若工作模式变更导致总工时下降，超限摸鱼时长必须自动修正

## 5.5 动力舱

- 每日任务次数不可超上限
- 每周成就只在周维度重置
- 等级阈值必须与文档一致
- 进度条边界值必须正确

## 5.6 跨模块联动

- 首页“领潮中心”修改今日摸鱼时长后：
  - 首页更新
  - 战报更新
  - 个人基地打卡天数更新
  - 动力舱相关预览更新
- 动力舱完成任务并导致等级变化后：
  - 个人基地头像外圈 tone 更新
  - 个人基地头像角标 icon 更新
  - 个人基地 `Lv` 与等级名文案更新
- 修改工作模式、上下班时间或午休规则后：
  - 相关自然日的摸鱼时长若超出新工时上限，必须先自动裁剪
  - 再统一刷新首页、战报、个人基地等所有派生视图

## 5.7 社区接入

- 主按钮必须拉起真实能力
- 常见错误码必须有明确提示
- 能力不可用、网络失败、群满员等场景必须有反馈

## 5.8 场景化验收样例

- 工作日 `08:30` 打开首页：
  - 顶部状态显示 `待启航`
  - 不允许开启摸鱼
  - 今日航行收益与今日摸鱼收益均为 `0`
- 工作日 `10:00` 开启摸鱼 `15` 分钟：
  - 今日摸鱼收益只按这 `15` 分钟计算
  - 今日航行收益只按截至 `10:15` 的已发生工时减摸鱼时长计算
  - 不得显示整日预计收益
- 工作日活跃摸鱼会话跨入午休：
  - 会话自动终止
  - 首页状态切为 `午休中`
  - 当天记录正确结算到午休开始时刻
- 打开航行报告最近记录区域：
  - 不出现编辑按钮
  - 不出现修改弹窗
  - 页面只展示读取后的真实记录
- 用户在首页“领潮中心”将今日摸鱼时长改为大于今日总工时的值：
  - 系统自动裁剪到今日总工时上限
  - 保存后首页与战报显示同一结果
- 用户修改月薪后：
  - 历史战报金额同步变化
  - 首页本月累计航行收益同步变化
- 用户修改工作制或午休时段后：
  - 历史月报中的工作/摸鱼占比同步变化
  - 日历工作日判定同步变化
  - 若某日原摸鱼时长超过新的日工时上限，该日摸鱼时长被自动修正后再参与计算
- 动力舱完成任务触发升级后：
  - 重新进入个人基地，头像徽章 tone、icon、`Lv` 等级文案与动力舱保持一致
- 清空数据后再次初始化：
  - 个人基地默认等级恢复为动力舱初始等级
  - 不得残留旧等级样式或旧等级文案
- 用户删除一张自建周年卡后：
  - 系统卡仍完整存在
  - 首页岁月坐标卡片重新排序
- 用户修改生日或入职日期后：
  - 对应系统卡日期立即更新
  - 首页岁月坐标与人生航程同步刷新
